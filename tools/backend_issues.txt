ISSUE #1: Missing "DateOfBirth" column in StagingPersons table
--------------------------------------------------------------
Severity: CRITICAL (blocks all package staging)

Description:
  The EF Core entity model "StagingPerson" has a property "DateOfBirth"
  (DateTime?), but the PostgreSQL table "StagingPersons" does NOT have
  this column. Only "YearOfBirth" (integer) exists.

  When staging any package, the backend generates an INSERT statement
  that includes "DateOfBirth", causing a 500 error:

  ERROR: column "DateOfBirth" of relation "StagingPersons" does not exist

Steps to reproduce:
  1. Upload any valid .uhc package via POST /api/v1/ImportPackages/upload
  2. Call POST /api/v1/ImportPackages/{id}/stage
  3. Backend fails with 500 Internal Server Error

Fix options:
  A) Add EF migration to create the missing column:
     ALTER TABLE "StagingPersons" ADD COLUMN "DateOfBirth" timestamp with time zone;

  B) Remove "DateOfBirth" property from EF Core entity if not needed
     (rely on "YearOfBirth" only)

Temporary workaround applied:
  Column added manually via ALTER TABLE (not in migrations)


ISSUE #2: Package stuck in "Staging" status after failed staging
----------------------------------------------------------------
Severity: HIGH (requires manual DB cleanup)

Description:
  When a staging operation fails (e.g., due to Issue #1), the backend
  changes the package status to "Staging" BEFORE the operation completes.
  If the operation fails, the status is NOT rolled back.

  Subsequent staging attempts return 409:
  "Cannot stage package with status 'Staging'. Expected Validating or ValidationFailed."

  The package becomes permanently stuck and cannot be re-staged without
  direct database intervention.

Steps to reproduce:
  1. Upload a .uhc package
  2. Attempt staging (fails due to Issue #1 or any other error)
  3. Try staging again -> 409 Conflict

Expected behavior:
  - On failure, roll back status to previous state (e.g., "Uploaded")
  - OR allow re-staging packages with status "Staging"

Current workaround:
  Manual cleanup: DELETE FROM "ImportPackages" WHERE "Status" = 'Staging';


ISSUE #3: InvalidCastException - EF Core int/varchar type mismatch
-------------------------------------------------------------------
Severity: CRITICAL (blocks viewing staged entities)

Description:
  The GET /api/v1/import/packages/{id}/staged-entities endpoint crashes
  with a 500 error:

  InvalidCastException: Reading as 'System.Int32' is not supported
  for fields having DataTypeName 'character varying'

  This happens in StagingRepository.GetByPackageIdAsync (line 69).

  Root cause: The EF Core entity model defines certain properties as
  'int' (likely Gender, Nationality, or similar vocab-code fields),
  but the corresponding PostgreSQL columns are 'character varying'.

  Evidence from database:
    StagingPersons."Gender" = varchar(20), stored value: "1"
    StagingPersons."Nationality" = varchar(100), stored value: "1"

  The backend staging handler converts vocabulary values to numeric
  codes and stores them as strings in varchar columns. But the EF Core
  model tries to read them back as int, causing the crash.

Steps to reproduce:
  1. Upload and stage a .uhc package (staging succeeds)
  2. Call GET /api/v1/import/packages/{id}/staged-entities
  3. Backend crashes with InvalidCastException

Stack trace location:
  StagingRepository`1.GetByPackageIdAsync()
  -> GetStagedEntitiesQueryHandler.Handle()

Fix:
  Align EF Core entity property types with database column types.
  If Gender/Nationality are varchar in DB, the C# properties should
  be string (not int). Or vice versa - change DB columns to integer.


ISSUE #4: Partial/silent staging failure - most records lost
-------------------------------------------------------------
Severity: HIGH

Description:
  When staging a .uhc package with 3 buildings, 6 units, and 10 persons,
  only 1 building and 1 person are actually staged. The rest are silently
  dropped with no error reported.

  Results after staging:
    Buildings: 1 of 3 staged (67% lost)
    Units:     0 of 6 staged (100% lost)
    Persons:   1 of 10 staged (90% lost)

  The validation report confirms these low counts (Step 2 shows
  Buildings=1, Persons=1, Units=0).

  Additionally, some fields are NULL that should have values:
    - StagingBuildings."NumberOfFloors" = NULL (should be integer)
    - StagingPersons."YearOfBirth" = NULL (should be integer)
    - StagingBuildings."Status" = 0 (we sent 1-5, default seems wrong)

  This suggests the staging handler has issues mapping .uhc SQLite
  column names to staging table columns for some fields.

Steps to reproduce:
  1. Generate a .uhc with multiple buildings/units/persons
  2. Upload and stage
  3. Query staging tables directly: SELECT COUNT(*) FROM "StagingBuildings"
  4. Observe most records are missing

Expected behavior:
  All records from .uhc should be staged (or explicitly reported as
  failed with error details).


ISSUE #5: Approve endpoint requires request body
--------------------------------------------------
Severity: MEDIUM (fixed on desktop side)

Description:
  POST /api/v1/import/packages/{id}/approve returns 400:
  "A non-empty request body is required. The command field is required."

  The endpoint expects a JSON body (e.g., {"packageId": "..."}) but
  the API documentation does not specify required body fields.

  Fixed on desktop client by sending {"packageId": package_id} as
  the request body.

Note:
  The commit endpoint (POST /api/v1/import/packages/{id}/commit) may
  have the same requirement - also fixed preventively on desktop side.


ISSUE #6: Staging handler does not map Status, NumberOfFloors, YearOfConstruction
----------------------------------------------------------------------------------
Severity: HIGH (data corruption - fields default to 0 or NULL)

Description:
  After staging a .uhc package, several fields in staging tables are NOT
  populated from the .uhc SQLite data:

    StagingBuildings."Status" = 0 (sent values 1-5 in .uhc, always defaults to 0)
    StagingBuildings."NumberOfFloors" = NULL (sent integer values in .uhc)
    StagingBuildings."YearOfConstruction" = NULL (sent integer values in .uhc)

  The staging handler appears to map some .uhc columns to staging table
  columns (e.g., BuildingId, GovernorateCode, Latitude, Longitude) but
  misses others.

  Possible cause: The .uhc uses snake_case column names (e.g., "status",
  "number_of_floors", "year_of_construction") and the backend PascalCase
  conversion may not handle all columns. Or these columns are simply not
  included in the staging handler's column mapping.

Steps to reproduce:
  1. Generate a .uhc with buildings having status=3, number_of_floors=5
  2. Upload and stage
  3. SELECT "Status", "NumberOfFloors", "YearOfConstruction"
     FROM "StagingBuildings"
  4. Observe Status=0, NumberOfFloors=NULL, YearOfConstruction=NULL

Expected behavior:
  All .uhc columns should be mapped to their corresponding staging table
  columns. At minimum: Status, NumberOfFloors, YearOfConstruction.

Impact:
  - Validation always fails with "Invalid BuildingStatus: 0"
  - Buildings cannot pass validation without manual DB update


ISSUE #7: PropertyUnits fail during commit (0 of 6 committed)
--------------------------------------------------------------
Severity: HIGH (units are lost during commit)

Description:
  After staging and approving a package with 6 property units, the commit
  endpoint returns PartiallyCompleted with units: 0 committed, 6 failed.

  Commit result:
    Buildings:     3 approved, 3 committed, 0 failed  (100% success)
    Persons:      10 approved, 10 committed, 0 failed  (100% success)
    PropertyUnits: 6 approved, 0 committed, 6 failed   (0% success)

  The buildings commit succeeds, but the property units that reference
  those buildings fail. This suggests an issue with the OriginalBuildingId
  UUID mapping during commit - the unit's OriginalBuildingId (UUID) may
  not resolve to the committed building's ID.

Steps to reproduce:
  1. Upload and stage a .uhc with buildings and units
  2. Force-approve all entities (set ValidationStatus=1, IsApprovedForCommit=true)
  3. Commit the package
  4. Observe commit report: units = 0 committed, all failed

Expected behavior:
  Property units should commit successfully when their parent buildings
  commit successfully.

Steps to reproduce:
  1. Upload and stage any .uhc package
  2. Call GET /api/v1/import/packages
  3. Observe buildingCount, propertyUnitCount, personCount are all 0

Expected behavior:
  Content counts should reflect the actual number of staged/committed
  entities.
